Fix the custom budget cycle so that "Tanggal ke Tanggal" periods persist correctly and are used consistently across the entire app (budget page, dashboard calculations, and labels).

Current problem:
The budget wizard correctly shows the custom cycle (example: "12 Mar – 11 Apr") in the confirmation step, but after saving the plan the budget page and dashboard revert to a calendar month ("01 Mar – 31 Mar").

This indicates two issues:

1. The cycle fields are not always persisted when updating an existing plan.
2. The backend computes the active cycle relative to today's date instead of anchoring it to the budget plan's month.

Required fixes:

---

1. Fix persistence in storage layer

File: storage.ts

Inside the `upsertBudgetPlan` update branch, include the fields:

* cycleType
* cycleStartDay

Example:

update({
income,
strategy,
allocations,
cycleType,
cycleStartDay,
updatedAt: new Date()
})

This ensures the wizard properly saves the custom cycle selection.

Do NOT change the database schema.

---

2. Fix cycle calculation logic

Currently the backend calculates the cycle relative to today's date.
Replace this with a plan-month anchored calculation.

Correct rule:

If cycleType === "monthly":
startDate = first day of plan month
endDate = last day of plan month

If cycleType === "date_to_date":
startDate = new Date(planYear, planMonth, cycleStartDay)

```
endDate = new Date(startDate)
endDate.setMonth(endDate.getMonth() + 1)
endDate.setDate(endDate.getDate() - 1)
```

Example:

plan month = March
cycleStartDay = 12

Result must always be:

12 Mar – 11 Apr

The calculation must NOT depend on today's date.

---

3. Fix dashboard cycle math

In FinancialSummaryCard or wherever the daily budget and remaining days are calculated, ensure:

daysPassed + remainingDays = daysInCycle

Remove any double "+1" additions that cause cycles to become 32 days.

Safe daily budget must be calculated as:

safeDaily = remainingBudget / remainingDays

Using the correct cycle range.

---

4. Ensure dashboard and budget calculations follow the selected cycle

The following values must use the computed cycle range:

* income detection
* expense totals
* remaining monthly budget
* safe daily budget
* progress bars
* "days left" calculations

Currently they still assume a calendar month (1-31).

---

5. UX improvement for cycle label

Improve the budget card UI so users clearly understand the custom cycle.

When cycleType === "date_to_date":

Show:

Periode Aktif
12 Mar – 11 Apr
Siklus Gaji

When cycleType === "monthly":

Show:

Periode Aktif
01 Mar – 31 Mar
Bulanan

Do not change layout height — only add the small subtitle text under the period label.

Font style:

* small
* muted opacity
* secondary text color

This helps users understand that the budget is based on their salary cycle.

---

Important constraints:

* Do NOT change business logic for allocations or budgeting strategy.
* Do NOT change existing API routes.
* Do NOT change database schema.
* Only fix persistence, cycle calculation, and UI label.

"12 Mar – 11 Apr"

Budget page:
Periode Aktif

calculated based on the same cycle.
